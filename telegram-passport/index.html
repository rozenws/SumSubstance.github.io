        # Telegram Passport

        This integration option is very easy. Just put a Telegram button on your page and that's it. Of course, a user will be asked to install Telegram if they haven't done so yet.

        We recommend using this option in combination with WebSDK, as users may not use the Telegram.

        # Preparations

        To initialize a Telegram Passport you should generate a temporary access token that is also passed to the Telegram Passport initialization. This action must be performed on a server side. Please also pass a userId parameter that is unique and meaningful. It can be an external user ID in your system, or an email address. Although they may be used for testing, please do not randomly generate these IDs.

        Do not reuse tokens. Create a new token every time you show a Telegram Passport. Each token has a lifespan, which you provide upon its creation. To avoid its expiration, just create a new one each time.

        If it is necessary for users to pass verification through Telegram Passport, there are some steps to follow:

        • Create ACCESS_TOKEN by request from your backend
        • Initialize the button as in this example:
        Name Description
        bot_id ID of the bot.
        scope List of documents to be requested from the user.
        public_key The public key to encrypt the data. Is a constant.
        payload After sending the data, the user will be redirected to this page.

        669660896 — the bot_id for the test environment

        691314081 — the bot_id for the product environment

        • Or you can add a button on your page as described here. You can also use Telegram Passport mobile SDKs to do the same. The important thing is to provide a generated token as payload (see the Preparations section).

        • After verification is finished, we will send the callback to your endpoint. (Getting verification results)

        # Important notes

        • When switching to production make sure to change the bot ID. The public key stays the same. ACCESS_TOKEN should be generated according to the environment.

        • When you generate a token you use the userId. Using these values, you can get the applicantId:

        GET /resources/applicants/-;externalUserId=${userId}
